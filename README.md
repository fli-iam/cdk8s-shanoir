cdk8s constructs for shanoir-ng
===============================

**CAUTION: this is a work in progress (not ready for production)**



## Getting started

### 0. prerequisites

Before starting you need to install:
- [cdk8s](https://cdk8s.io/docs/latest/get-started/)
- [npm](https://www.npmjs.com/package/npm)

### 1. initialise a new cdk project

```sh
mkdir shanoir-example
cd shanoir-example/
cdk8s init typescript-app
```

### 2. add the `cdk8s-shanoir` package to your project

```sh
npm install github:fli-iam/cdk8s-shanoir
```

### 3. configure a new shanoir instance

In your `main.ts`, create a `ShanoirNGChart` object for your shanoir instance:

```ts
import { App, Size } from 'cdk8s';
import { PersistentVolumeClaimProps, PersistentVolumeAccessMode } from 'cdk8s-plus-33'
import { ShanoirNGChart, shanoirVolumes } from 'cdk8s-shanoir';


const app = new App();

function pvc(mb: number): PersistentVolumeClaimProps {
  return { storage: Size.mebibytes(mb), accessModes: [PersistentVolumeAccessMode.READ_WRITE_ONCE] };
}

new ShanoirNGChart(app, 'shanoir-example',
{
  // URLs served by shanoir (main UI and DICOM viewer)
  url: "https://shanoir.example.org",
  viewerUrl: "https://shanoir-viewer.example.org",

  // Name and email (for signing outgoing emails)
  adminName: "Example Admin",
  adminEmail: "admin@example.org",

  // (optional) Name of this shanoir instance + css color
  instanceName: "Example",
  instanceColor: "red",

  // SMTP parameters for outgoing emails
  smtp: {
    host: "stmp.example.org",
    auth: { username: "smtp-user", password: "stmp-pass" },
    fromAddress: "no-reply@example.org",
  },

  // Name and password of the keycloak admin account (in the master realm)
  keycloakCredentials: { username: "admin", password: "qzflpuy;" },

  // Persistent volume claims
  volumeClaims: {
    // medical data
    "datasets-data":              pvc(64),
    "dcm4chee-arc-storage-data":  pvc(512),
    "extra-data":                 pvc(1),
    "studies-data":               pvc(2),

    // databases
    "database-data":              pvc(2),
    "dcm4chee-database-data":     pvc(4),
    "keycloak-database-data":     pvc(1),

    // logs
    "logs":                       pvc(2),
    "dcm4chee-logs":              pvc(8),
    "keycloak-logs":              pvc(1),

    // disposable data (temporary data, indexes, ...)
    "rabbitmq-data":              pvc(1),
    "solr-data":                  pvc(1),
    "tmp":                        pvc(32),
  },

  // Extra config needed for the first run
  //
  // initialisation mode, for the first run (CAUTION: THE DATABASES WILL BE WIPED OUT)
  init: true,
  ingress: {
    // expose the keycloak console at `${url}/auth/admin/` (needed to create the first users)
    exposeKeycloakAdminConsole: true,
  },

  // (optional) disable the hash suffix object names generated by cdk8s
  disableResourceNameHashes: true,
});

app.synth();
```

Notes:
- The full list of options is [documented in the `ShanoirNGProps`class](src/shanoir-ng-props.ts).

- for a development instance, you may use the following SMTP config:
  ```ts
  smtp: {
    mailpit: { host: "mailpit.example.org" },
    fromAddress: "no-reply@example.org",
  },
  ```
  This will deploy a [mailpit](https://mailpit.axllent.org/) container to collect all outgoing mails
  and make them browsable at http://mailpit.example.org

- `init: true` will be used only for the initial run so as to execute the initial migrations.
  <br>**CAUTION**: `init: true` should not be used on an existing instance because it will wipe out
  the databases.

- By default the generated ingress object do not include rules for the keycloak master realm and
  admin console (it only exposes the shanoir-ng realm). They can be included by setting:
  ```ts
  ingress: { exposeKeycloakAdminConsole: true }
  ```
  (this will be needed for activating the initial user accounts).


### 4. generate your manifest

Run the command:
```sh
cdk8s synth
```

This will generate the kubernetes manifests for your shanoir instance in `dist/shanoir-example.k8s.yaml`.

When running in the *initialisation* mode, the manifests implementing destructive actions are stored
in a separate file `dist/danger-init-shanoir-example.k8s.yaml` to avoid any mishaps.

### 5. first run (initial migrations)

The manifests **must** be generated with **`init: true`**.

Deploy the main manifests:
```
kubectl apply -f dist/shanoir-example.k8s.yaml
```

Wait until all deployments are up-to-date:
```
$ kubectl get deploy -w
NAMESPACE        NAME                                      READY  UP-TO-DATE  AVAILABLE  AGE
shanoir-example  shanoir-example-database-deploy           1/1    1           1          4m37s
shanoir-example  shanoir-example-dcm4chee-database-deploy  1/1    1           1          4m37s
shanoir-example  shanoir-example-dcm4chee-deploy           1/1    1           1          4m36s
shanoir-example  shanoir-example-keycloak-database-deploy  1/1    1           1          4m37s
shanoir-example  shanoir-example-mailpit-deploy            1/1    1           1          4m37s
shanoir-example  shanoir-example-rabbitmq-deploy           1/1    1           1          4m37s
shanoir-example  shanoir-example-solr-deploy               1/1    1           1          4m37s
```

Deploy the initialisation manifests, this will create a deployment for keycloak and a job for the
shanoir microservices:
```
$ kubectl apply -f dist/danger-init-shanoir-example.k8s.yaml
```

Wait until the keycloak deployment is ready (this may take a few minutes):
```
$ kubectl get pod -w
NAME                                                           READY   STATUS            RESTARTS   AGE
danger-init-shanoir-example-keycloak-deploy-647c654b5c-nmcfc   0/1     PodInitializing   0          97s
...
danger-init-shanoir-example-keycloak-deploy-647c654b5c-nmcfc   1/1     Running           0          98s
```

When it is ready, you may check its output to ensure that it is listening:
```
$ kubectl logs danger-init-shanoir-example-keycloak-deploy-f6c4frhh4h
...
Keycloak 26.2.1 on JVM (powered by Quarkus 3.20.0) started in 6.129s. Listening on: http://0.0.0.0:8080
```

As long as issue [#16](https://github.com/fli-iam/cdk8s-shanoir/issues/16) is not fixed, the job for
initialising the microservices does not correctly run because it needs the keycloak service to be
available (so as to synchronise the initial users in the *shanoir-ng* realm).

Delete the *ms* job and re-apply the initialisation manifests:
```
$ kubectl delete job/danger-init-shanoir-example-ms-job
job.batch "danger-init-shanoir-example-ms-job" deleted

$ kubectl apply -f dist/danger-init-shanoir-example.k8s.yaml
deployment.apps/danger-init-shanoir-example-keycloak-deploy-c818faf7 configured
job.batch/danger-init-shanoir-example-ms-job-c880db34 created
```

Wait until the job is complete:
```
$ kubectl get job -w
NAME                                 STATUS               COMPLETIONS   DURATION   AGE
danger-init-shanoir-example-ms-job   Running              0/1           34s        34s
danger-init-shanoir-example-ms-job   SuccessCriteriaMet   0/1           77s        77s
danger-init-shanoir-example-ms-job   Complete             1/1           77s        77s
```

Check the output of the containers to ensure there are no errors:
```
$ kubectl get pods
NAME                                          READY   STATUS      RESTARTS   AGE
danger-init-shanoir-example-ms-job-pjrhd      0/5     Completed   0          2m43s
...

$ kubectl logs --all-containers danger-init-shanoir-example-ms-job-pjrhd
```

Open the keycloak admin console at: <https://shanoir.example.org/auth/admin> with your web browser
and sign in using the master admin account from your config

Select **Manage Realms** in the left panel, then **shanoir-ng** so as to make it the current
realm.

Select **Users** in the left panel, then **dummy-admin** so as to select the initial admin user.

Go to the **credentials** tab and select **Reset password** to set a new password for this user.


Delete all api objects from the initialisation manifests file:
```
$ kubectl delete -f dist/danger-init-shanoir-example.k8s.yaml
```

The initialisation is complete, you can now remove the `init: true` parameter and regenerate your
manifests with `cdk8s synth`.


### 6. normal run

The manifests **must** be generated with **`init: false`**.

Deploy the shanoir manifests and wait until all deployments are ready:
```
$ kubectl apply -f dist/shanoir-example.k8s.yaml

$ kubectl get deploy -w
```

You may now use shanoir at https://shanoir.example.org/ and sign in as the *dummy-admin* user.
